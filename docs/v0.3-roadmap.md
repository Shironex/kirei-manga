# KireiManga v0.3 — Translation Engine

Tracks the work for v0.3 (Translation Engine) as defined in `docs/kireimanga-prd.md` §5.3.
v0.2 (Local Library) is the prerequisite — the translation overlay runs on
top of any reader page, but having local raw JP manga is the primary motivator
so that slice ships first.

This is the headline feature of KireiManga and the most technically involved
milestone: a C++ OpenCV native addon for bubble detection, a Python sidecar
for manga-OCR, a translation provider layer, a SQLite cache keyed on page
hash + bubble box, and an HTML canvas overlay that renders translated text
sized to fit each detected bubble.

## Cross-cutting rules

- `P:\kirei-manga` (Windows) / `~/dev/kirei-manga` (macOS) — same repo, `master` branch.
- Small conventional commits per phase — no batch commits.
- No `Co-Authored-By` on any commit.
- Everything except the translation API call runs locally. A user with no internet + Ollama installed should be able to read translated manga.
- Bubble detection runs in the C++ worker thread. It must NEVER block the Node event loop — use `Napi::AsyncWorker`.
- The Python sidecar is a black-box child process. If it crashes, the translation feature degrades gracefully (`Tesseract` fallback if available, otherwise feature disabled with a clear error — rest of the app keeps working).
- All translation state is per-user, per-machine. No cloud sync in v0.3.
- Overlay rendering happens in the renderer on a `<canvas>` sized to the image's natural dimensions. The main process never rasterizes text.
- DeepL / Google API keys are stored via `electron-store` (plain — not OS keychain for v0.3; revisit in v0.5). Never logged, never sent over the socket to the renderer in plain form.
- Translation provider failures don't corrupt the cache — only successful translations are persisted.
- Design identity per `.impeccable.md`: overlay text uses the same Fraunces / Shippori Mincho / Geist palette. Translated text boxes use a soft `kinari` (paper white) background with `sumi` (deep ink) text — never a generic white rectangle.

## Slice A — Shared types for the pipeline

**Goal:** lock down the types the native addon, sidecar, provider layer, cache,
and overlay will all consume. Some already exist (`BoundingBox`,
`TranslationCache`, `OcrResult`) — extend rather than duplicate.

### Phases

1. **`feat(shared): extend translation pipeline types`**
   - `OcrResult` (already exists) stays `{ boxIndex, text }`.
   - Add `BubbleDetectionResult` → `{ boxes: BoundingBox[], imageWidth: number, imageHeight: number, durationMs: number }`.
   - Add `PageTranslation` → `{ pageHash: string, bubbles: Array<{ box: BoundingBox; original: string; translated: string; provider: TranslationProviderId; targetLang: string }> }`.
   - Add `TranslationProviderId = 'deepl' | 'google' | 'ollama' | 'tesseract-only'` (tesseract is OCR-only, never a full provider — keeps the union closed).
   - Add `TranslationProviderStatus` → `{ id; ok: boolean; reason?: string; remainingChars?: number }` for the settings UI.

2. **`feat(shared): add translation channel payloads`**
   - In `ipc-payloads.ts` extend the existing `translation:*` interfaces, plus add:
     - `TranslationGetPagePayload/Response` (page-hash-based lookup).
     - `TranslationRunPipelinePayload` (pageImagePath + targetLang + provider hint).
     - `TranslationRunPipelineResponse` (`PageTranslation`).
     - `TranslationProviderStatusResponse`.
     - `TranslationReportBadPayload/Response` (Slice L).

3. **`feat(shared): add per-series translation settings type`**
   - Extend `AppSettings.translation` with `TranslationSettings`:
     - `enabled: boolean`, `defaultProvider: TranslationProviderId`, `targetLang: string`, `autoTranslate: boolean`, `overlayFont: string`, `overlayOpacity: number (0–1)`, `providerKeys: { deepl?: string; google?: string; ollamaEndpoint?: string }`.
   - Add `Series.translationOverride?: Partial<TranslationSettings>` for per-series override.

4. **`feat(desktop): migration 007 for translation_cache + series override`**
   - `CREATE TABLE translation_cache (id TEXT PRIMARY KEY, page_hash TEXT NOT NULL, bubble_index INTEGER NOT NULL, bbox_x INTEGER NOT NULL, bbox_y INTEGER NOT NULL, bbox_w INTEGER NOT NULL, bbox_h INTEGER NOT NULL, original_text TEXT NOT NULL, translated_text TEXT NOT NULL, target_language TEXT NOT NULL, provider TEXT NOT NULL, created_at TEXT NOT NULL DEFAULT (datetime('now')), UNIQUE(page_hash, bubble_index, target_language, provider));`
   - Index on `(page_hash, target_language, provider)` — read-hot lookup pattern.
   - `ALTER TABLE series ADD COLUMN translation_override TEXT` (stored as JSON).

## Slice B — C++ bubble-detector native addon

**Goal:** `native/bubble-detector/` produces a prebuilt `.node` addon that
exposes `detectBubbles(imagePath): Promise<BoundingBox[]>` backed by OpenCV.
The scaffold folder already exists — this slice fills it in and wires it to
the desktop build.

### Phases

1. **`chore(native): add opencv build script + binding.gyp opencv link paths`**
   - **Decision (recorded in `native/bubble-detector/README.md`):** Build OpenCV from source in CI, link static, ship per-platform `.node` via `prebuildify` + `node-gyp-build` under `prebuilds/`. Pinned 4.11.x, minimal modules (core/imgproc/imgcodecs).
   - Phase 1 was split into 1a (this docs commit) + 1b (build script + binding.gyp wiring).

2. **`feat(native): implement bubble detector core`**
   - `src/detector.cpp` (C++17):
     - Load image via `cv::imread(path, cv::IMREAD_GRAYSCALE)`.
     - `cv::adaptiveThreshold(src, bin, 255, ADAPTIVE_THRESH_GAUSSIAN_C, THRESH_BINARY_INV, blockSize, C)`.
     - `cv::findContours(bin, contours, RETR_EXTERNAL, CHAIN_APPROX_SIMPLE)`.
     - Filter contours by area (min `(imgArea * 0.002)`, max `(imgArea * 0.3)`), bounding-rect aspect ratio (0.2–5), convexity (≥ 0.85), solidity (≥ 0.8).
     - Sort by `y, x` (natural reading order — Slice C refines for RTL).
     - Return array of `{ x, y, w, h, confidence }` via Napi.

3. **`feat(native): expose detector as AsyncWorker`**
   - `DetectBubbles(info)` returns a Promise. `Napi::AsyncWorker` subclass does the OpenCV work on the worker pool; `OnOK` converts the result to a JS array.

4. **`chore(native): prebuild scripts + CI matrix`**
   - `scripts/native-build.mjs` already exists — extend to cover the new addon if not already covered.
   - GitHub Actions: `windows-latest` + `macos-latest` prebuild jobs. Upload artifacts to `prebuilds/{platform}-{arch}/`. `postinstall` copies from here.

5. **`feat(desktop): wire native addon into translation module`**
   - `modules/translation/bubble-detector.service.ts` — thin wrapper around the addon. If the addon fails to load at boot, log a warning and mark the provider as unhealthy (translation feature disables gracefully).

6. **`test(native): detector detects bubbles on fixture pages`**
   - Unit tests in `native/bubble-detector/test/detector_test.cpp` (GoogleTest, built via a parallel `test/CMakeLists.txt` that links the same vendored OpenCV install as `binding.gyp`). Path is `test/`, not `src/`, so binding.gyp's source list isn't polluted by `_test.cpp`.
   - Node-level smoke test in `apps/desktop/src/modules/translation/bubble-detector.smoke.spec.ts` runs against the **real** addon (auto-skips when no prebuild) using a tiny committed PNG fixture under `apps/desktop/test/fixtures/translation/`.

### Non-goals in B

- Text orientation detection — Slice C tuning.
- OCR — Slice D.

## Slice C — Bubble-detector tuning

**Goal:** reduce false positives (screentone bubbles, panel borders) and handle
the common failure cases. Ship-quality = ~90% recall, <5% false-positive rate
on a hand-labeled fixture set of 20 pages across 5 series.

### Phases

1. **`feat(native): ignore screentone-dense regions`**
   - After threshold, compute local variance over each contour's bounding box in the grayscale source. Skip contours where variance falls between two thresholds characteristic of dotted screentone.

2. **`feat(native): drop panel-border contours`**
   - Panel borders are long, thin, axis-aligned. Drop contours where `min(w, h) / max(w, h) < 0.1`.

3. **`feat(native): RTL-aware sort option`**
   - `detectBubbles(imagePath, { direction: 'rtl' | 'ltr' })`. Sort order in reading direction.

4. **`feat(desktop): fixture page benchmark`**
   - `scripts/bench-bubble-detector.mjs` runs the detector across a fixture folder, prints recall/precision against a human-labeled JSON.

### Non-goals in C

- Per-panel detection. Bubbles-only is the scope.

## Slice D — manga-ocr Python sidecar

**Goal:** bundle the `manga-ocr` ML model as a single PyInstaller binary
shipped in `resources/sidecar/`. The desktop spawns it on startup, pipes JSON
requests over stdin, parses JSON responses over stdout.

### Phases

1. **`feat(sidecar): ocr server script`**
   - `sidecar/manga-ocr/main.py`: loads `manga-ocr` model once, reads `{ "image_path": "...", "boxes": [...] }` newline-delimited from stdin, for each box crops the image and runs OCR, emits `{ "results": [{ "box_index": i, "text": "..." }] }`.
   - Handles `{"op": "ping"}` for healthcheck.
   - Graceful shutdown on `{"op": "shutdown"}` or SIGTERM.

2. **`build(sidecar): PyInstaller bundle script`**
   - `sidecar/manga-ocr/build.py`: invokes PyInstaller with `--onefile --name kirei-ocr` and copies the result to `resources/sidecar/{platform}/kirei-ocr(.exe)?`.
   - `.gitignore` the output — it's built by CI.

3. **`ci: build sidecar in GitHub Actions matrix`**
   - Windows + macOS build jobs. Artifact size sanity check (manga-ocr is ~450MB; expect a big binary — document this in the installer size open question). Decision recorded: bundle ships via GitHub Release asset; desktop downloads on first translation request (Slice D.4). Asset URL convention: `kirei-ocr-{platform}-{arch}.tar.gz` per release tag.

4. **`feat(desktop): sidecar process manager`**
   - `modules/translation/sidecar/ocr-sidecar.service.ts`:
     - Spawns the bundled binary from `resources/sidecar/` on app boot (lazy — on first translation request).
     - Line-delimited JSON over stdin/stdout with a correlation id per request.
     - 60s timeout per request; queue with max-concurrency 1 (OCR is GPU/CPU-bound; don't stack).
     - Auto-restart on exit with exponential backoff (max 3 tries, then mark unhealthy).

5. **`feat(desktop): sidecar healthcheck + status channel`**
   - `translation:provider-status` gateway message returns `TranslationProviderStatus` for each provider. The renderer uses this to decide whether to show the translate button enabled.

6. **`test(desktop): sidecar manager handles crash and restart`**
   - Mock the child process; assert reconnection flow.

### Non-goals in D

- Shipping the Python interpreter separately — everything is baked into the PyInstaller bundle.

## Slice E — Translation provider abstraction + DeepL

**Goal:** a `TranslationProvider` interface the rest of the pipeline depends on;
concrete DeepL implementation with API-key storage, usage accounting, and
rate-limit handling.

### Phases

1. **`feat(desktop): translation provider interface`**
   - `modules/translation/providers/provider.interface.ts`:
     ```ts
     interface TranslationProvider {
       id: TranslationProviderId;
       translate(texts: string[], targetLang: string): Promise<string[]>;
       status(): Promise<TranslationProviderStatus>;
     }
     ```

2. **`feat(desktop): DeepL provider`**
   - Uses `net.fetch` (same rationale as MangaDex client). Free API endpoint: `https://api-free.deepl.com/v2/translate`. Detects pro keys by prefix and switches endpoint.
   - Batches up to 50 texts per request. Rate-limit: respects `Retry-After`. Logs usage from `https://api-free.deepl.com/v2/usage`.
   - API key sourced from `AppSettings.translation.providerKeys.deepl`. If missing, `status()` returns `{ ok: false, reason: 'no-api-key' }`.

3. **`feat(desktop): provider registry + pick-by-hint`**
   - `TranslationProviderRegistry` holds all providers. `pickProvider(hint?)` returns the first healthy provider in preference order (hint > default > any-healthy).

4. **`test(desktop): DeepL provider batches and retries`**
   - Mock `net.fetch`. Assert batching, 429 retry, key-missing status.

### Non-goals in E

- Google Translate (Slice I), Ollama (Slice J).

## Slice F — Page hash + translation SQLite cache

**Goal:** once a page's bubbles are OCR'd and translated, the result is
persisted so re-opening the same page is instant and offline.

### Phases

1. **`feat(desktop): page-hash utility`**
   - `modules/translation/cache/page-hash.ts`: streams the image file through SHA-256, returns hex. Used as `page_hash` in the cache row.

2. **`feat(desktop): TranslationCacheService`**
   - CRUD over `translation_cache` table. Keys on `(page_hash, bubble_index, target_language, provider)`.
   - `getForPage(pageHash, targetLang, provider): PageTranslation | null`.
   - `putBubble(...)` per bubble (small writes; pipeline sequences them so partial results survive a provider crash).

3. **`feat(desktop): pipeline orchestrator`**
   - `modules/translation/translation.service.ts`:
     - `runPipeline(pageImagePath, targetLang, providerHint?)`:
       1. Hash page. 2. Cache hit? return. 3. Detect bubbles. 4. OCR. 5. Translate. 6. Cache each bubble. 7. Return `PageTranslation`.
     - Steps 3–5 happen in parallel where safe (bubble detection + OCR are serial per page; translation is a batch call afterwards).

4. **`feat(desktop): TranslationGateway`**
   - `translation:run-pipeline`, `translation:get-page`, `translation:provider-status`. Reuse `handleGatewayRequest`.

5. **`test(desktop): pipeline uses cache on second call`**

### Non-goals in F

- Cache portability (open question).

## Slice G — Overlay canvas + text fitting UI

**Goal:** the reader page gets a translucent overlay layer; translated text
appears anchored to each bubble's bounding box, auto-shrinking to fit.

### Phases

1. **`feat(web): translation hook for current page`**
   - `useTranslationForPage(pageUrl)` — calls `translation:get-page` first (cache hit), falls back to `translation:run-pipeline` when `autoTranslate` is on.

2. **`feat(web): overlay canvas renderer`**
   - `components/reader/TranslationOverlay.tsx` — absolutely-positioned layer matching the page image's bounding rect. Each bubble gets a `div` with `position: absolute; left: x; top: y; width: w; height: h;` and a `TextBox` child that binary-searches font-size to fit the translated string inside the box.

3. **`feat(web): toggle — original / translated / both`**
   - Keyboard: `T` cycles through the three modes. Keyboard cheat sheet in settings updated.
   - "Both" mode splits the bubble horizontally: original on top 40%, translated below — or stacks them if the bubble is wide.

4. **`feat(web): click a bubble to see original text popover`**
   - Popover shows the original JP text in Shippori Mincho, plus a "Copy" button.

5. **`feat(web): reader status pill while pipeline runs`**
   - Small indicator: "Translating page 4…" with a spinner that uses the existing editorial spinner motif.

6. **`feat(web): reader gracefully shows untranslated page on pipeline error`**
   - Error banner ("Translation unavailable — OCR sidecar offline") dismissible; page reads normally underneath.

### Non-goals in G

- Manual bubble re-positioning (v0.5+).

## Slice J — Ollama local-model provider

**Goal:** fully offline translation path using a user-installed Ollama instance.
Lower quality but zero API cost and no data leaves the machine.

### Phases

1. **`feat(desktop): OllamaProvider`**
   - Default endpoint: `http://localhost:11434`. Configurable.
   - Model: `aya` or `qwen2:7b` (document in settings — users can override).
   - Prompt template: system message instructing JP→target-lang translation, keep honorifics, no paraphrasing, one line per bubble.

2. **`feat(web): Ollama endpoint + model settings`**
   - Endpoint URL, model name, "Test connection" button.

3. **`feat(desktop): Ollama streaming opt-in`** *(deferred — batch-and-wait shipped in J.1; streaming is a v0.4 polish if users request it)*
   - Ollama supports streaming. For the pipeline we can either stream bubble-by-bubble into the overlay as they complete (good UX) or batch-and-wait (simpler). J.1 ships batch-and-wait (`stream: false` in `/api/chat` requests). Streaming opt-in revisited post-v0.3 if users ask for it.

### Non-goals in J

- Bundling Ollama or a model with the installer — Ollama must be installed separately (documented in the setup flow).

## Slice K — Tesseract JP fallback

**Goal:** if the manga-ocr sidecar is unavailable, fall back to Tesseract with
the `jpn_vert` + `jpn` models. Lower accuracy for stylized manga fonts but
better than nothing.

### Phases

1. **`feat(desktop): bundle tesseract traineddata`**
   - `resources/tesseract/jpn.traineddata` + `jpn_vert.traineddata`. Loaded via `node-tesseract-ocr` or a direct wasm binding.

2. **`feat(desktop): fallback ocr provider`**
   - `TesseractOcrProvider` implements an OCR-only subset of the provider interface (sidecar is the other implementer). Pipeline orchestrator picks the first healthy OCR backend.

3. **`feat(web): surface fallback in settings`**
   - Small row under manga-ocr status: "Fallback: Tesseract (active)" when the sidecar is down.

### Non-goals in K

- Shipping manga-specific Tesseract models (there aren't well-maintained ones).

## Open questions

- **Installer size (resolved Slice D.3):** ~450MB sidecar ships via GitHub Release asset, downloaded on first translation request with progress UI (Slice D.4). Initial installer stays under 200MB.
- **OpenCV distribution (resolved Slice B.1a):** Build from source in CI, prebuildify per-platform .node, no in-repo binaries. See native/bubble-detector/README.md.
- **Cache portability:** keep translation_cache in app-data SQLite (default, simple) or per-series alongside the local chapter files? v0.3 ships with app-data; portability is a v0.5 polish.
- **Overlay font fitting:** binary-search the font-size per bubble, or use `canvas.measureText` with a precomputed scale table? Binary-search is simpler and good enough at 60fps.
- **Languages beyond EN/PL:** the providers support many, but overlay font selection matters (CJK targets need a CJK font). Start with EN/PL/JP; document how to add more in a follow-up.
- **Auto-translate default:** on or off? Off is safer (surprise API-key burn). Ship off; prompt the user on first reader-open of a JP-language chapter.
- **macOS native-addon CI:** `scripts/native-build.mjs` currently early-exits on macOS, and `release.yml` only invokes the addon build as part of the electron-builder packaging step — there is no separate `prebuilds/{platform}-{arch}/` artifact pipeline. Slice B phase 4 must add macOS to the build matrix and decide whether prebuilt `.node` files are committed (LFS) or downloaded from a GitHub release asset on `postinstall`. Blocks Slice B phase 4 only — Slices A, D, E, F, H can proceed in parallel.
- **MangaDex-source series have no edit drawer:** Slice H phase 2 says "reuses the drawer from v0.2 Slice J" — that drawer (`apps/web/src/components/local/LocalMetadataDrawer.tsx`) is local-only. MangaDex-source series are edited from a full `SeriesDetail.tsx` page, not a drawer. Decide before Slice H: (a) build a parallel translation-override drawer for MangaDex series, (b) add a translation panel inline to `SeriesDetail.tsx`, or (c) ship overrides for local series only in v0.3 and revisit MangaDex in v0.4.
