# KireiManga v0.2 — Local Library

Tracks the work for v0.2 (Local Library) as defined in `docs/kireimanga-prd.md` §5.2.
v0.1 (MangaDex Reader) has shipped — this milestone adds first-class support
for user-provided local manga (folders, CBZ, CBR, ZIP) alongside the existing
MangaDex library, reusing the same reader, series detail, progress, bookmarks,
and settings UI.

## Status

- [x] Slice A — Shared types + DB schema for local series / archives
- [x] Slice B — Archive readers (CBZ, ZIP, image folder)
- [ ] Slice C — CBR support (deferred — see open questions)
- [x] Slice D — Folder scanner (detect series + chapters recursively)
- [x] Slice E — Import flow (pick folder → review → commit to library)
- [x] Slice F — `kirei-page://local/...` protocol branch + cover extraction
- [x] Slice G — Local series detail page + chapter list
- [x] Slice H — Local reader integration (page list, reading progress)
- [x] Slice I — Unified library view (local + MangaDex together)
- [x] Slice J — Manual metadata editor
- [x] Slice K — MangaDex metadata match
- [x] Slice L — Update checks for folders (detect new chapters on disk)

## Cross-cutting rules

- `P:\kirei-manga` (Windows) / `~/dev/kirei-manga` (macOS) — same repo, `master` branch.
- **Never** run `pnpm dev` for desktop/api unsolicited (global rule). Verify with `pnpm typecheck`, `pnpm build`, `pnpm test`.
- Small conventional commits per phase — no batch commits.
- No `Co-Authored-By` on any commit.
- Reuse existing reader / library / settings UI wherever possible. Source discrimination happens at the gateway layer (`series.source === 'local'`) and at the protocol layer (`kirei-page://local/...` vs `kirei-page://mangadex/...`).
- Never emit raw absolute filesystem paths to the renderer. All local images served via `kirei-cover://local/...` and `kirei-page://local/...`. Local ids in URLs are opaque UUIDs — never a user-visible path segment.
- Archive extraction runs in the main process, never the renderer. Extracted page bytes stream back through the custom protocol.
- Local chapter id = local SQLite `chapters.id` (UUID). The existing `/reader/:chapterId` route handles both sources uniformly — the reader page branches on `series.source` to pick the right page-list hook.
- Rate-limit-free path: local library operations are purely filesystem + SQLite; no upstream throttling applies. Keep scan I/O bounded (parallelism limit, cancellable) so it doesn't starve the event loop.
- Design identity per `.impeccable.md`: editorial, hairline rules, Fraunces display, bengara accent. Same visual language as v0.1 — the user should not notice a tonal shift between MangaDex and local series.

## Slice A — Shared types + DB schema for local series / archives

**Goal:** define the payload types the rest of the milestone will consume, and
extend the SQLite schema to store the bits we need (archive path, archive
format, content hash for dedup, root path for rescans).

### Phases

1. **`feat(shared): add local library types`**
   - Add `LocalArchiveFormat = 'folder' | 'cbz' | 'cbr' | 'zip'` to `packages/shared/src/types/series.ts`.
   - Extend `Chapter` with `localArchiveFormat?: LocalArchiveFormat` and keep `localPath` (already present) — for folders it points at a directory, for archives at the archive file.
   - Add `LocalSeriesMetaPatch` (title, titleJapanese, notes, score, status, coverPath) for the manual editor in Slice J.
   - Add scanner types in a new `packages/shared/src/types/local.ts`:
     - `ScanProgress` (phase: `'scanning' | 'reading-archives' | 'done'`, current, total, currentPath).
     - `ScanCandidateChapter` (relativePath, chapterNumber?, volumeNumber?, pageCount, format).
     - `ScanCandidateSeries` (absolutePath, suggestedTitle, coverCandidatePath?, chapters).
     - `ScanResult` (rootPath, scannedAt, candidates).

2. **`feat(shared): add local library channel payloads`**
   - In `ipc-payloads.ts` add:
     - `LocalScanPayload` / `LocalScanResponse`.
     - `LocalImportPayload` (candidates user confirmed) / `LocalImportResponse` (created series ids).
     - `LocalGetPagesPayload` (localChapterId) / `LocalGetPagesResponse` (pages: kirei-page URLs).
     - `LocalUpdateSeriesPayload` / `LocalUpdateSeriesResponse`.
     - `LocalRescanSeriesPayload` / `LocalRescanSeriesResponse` (for Slice L).
     - `LocalDeleteSeriesPayload` / `LocalDeleteSeriesResponse`.
   - Add `LocalEvents` enum to `constants/ipc-events.ts` matching the channel names.

3. **`feat(desktop): migration 006 for local library columns`**
   - `ALTER TABLE series ADD COLUMN local_root_path TEXT` — absolute path to the series folder (rescans read from here).
   - `ALTER TABLE series ADD COLUMN local_content_hash TEXT` — SHA1 of sorted relative chapter paths (dedup key).
   - `ALTER TABLE chapters ADD COLUMN local_archive_format TEXT` with the CHECK matching `LocalArchiveFormat`.
   - `CREATE INDEX idx_series_local_root ON series(local_root_path)` for rescan lookups.

4. **`test(desktop): migration 006 upgrades cleanly`**
   - Re-use the existing sql.js migration harness. Insert a v5 row, run migrations, assert the new columns exist and default to NULL.

### Non-goals in A

- Actually scanning or extracting archives (Slices B–D).
- Importing anything into the DB (Slice E).

## Slice B — Archive readers (CBZ, ZIP, image folder)

**Goal:** given a local path and optional archive format, return the ordered
list of image entries and stream a specific page's bytes on demand. This is
the foundation the protocol (Slice F) and the scanner (Slice D) both consume.

### Phases

1. **`chore(desktop): add node-stream-zip and image-size dependencies`**
   - `node-stream-zip` for CBZ/ZIP streaming (MIT, no native code).
   - `image-size` (sync buffer variant) for page dimension sniffing without decoding.
   - `mime-types` for content-type mapping on the protocol response.

2. **`feat(desktop): add archive-reader module`**
   - `modules/local/archive/archive-reader.ts` exports an interface:
     ```ts
     interface ArchiveReader {
       listPages(): Promise<PageEntry[]>;  // sorted natural order
       readPage(entry: PageEntry): Promise<{ data: Buffer; mime: string }>;
       close(): Promise<void>;
     }
     ```
   - `ZipArchiveReader` (CBZ + ZIP): wraps node-stream-zip. Lists entries filtered to image extensions (jpg/jpeg/png/webp/avif), sorts via a natural-sort comparator that handles `001.jpg`, `p01.png`, `01 - title.jpg` correctly.
   - `FolderArchiveReader`: reads directory entries, same filter + sort.
   - Natural sort: export as `naturalPageSort(a: string, b: string): number` so tests can hit it directly.

3. **`test(desktop): archive reader handles ordering and image filter`**
   - Unit tests with fixture folders and a hand-built zip (use node-stream-zip's writer).
   - Cases: `page_1.jpg`, `page_10.jpg`, `page_2.jpg` sort to 1, 2, 10.
   - Cases: non-image entries (e.g. `Thumbs.db`, `info.txt`) dropped.
   - Cases: mixed case extensions and leading dot-files ignored.

4. **`feat(desktop): add archive-reader factory`**
   - `openArchive(absolutePath, format): Promise<ArchiveReader>` — infers format by extension when format is `'folder'` + a dir, or by archive signature bytes otherwise. Rejects on unsupported formats (CBR lands in Slice C).

### Non-goals in B

- CBR support (Slice C).
- Any caching — caching lives at the protocol layer.

## Slice C — CBR support (optional)

**Goal:** handle CBR (RAR) archives. RAR is a non-open format; bundling an
unrar implementation is risky (licensing, prebuilds). Decision deferred to the
open questions — ship B and D without CBR first, revisit here.

### Phases

1. **`chore(desktop): evaluate unrar options`**
   - Document in the roadmap which option wins: `unrar-js` (pure JS, slow), `node-unrar-js` (wasm, faster), `node-rar` (native, fastest but prebuild pain), or "prompt user to convert to CBZ on import".
   - Spike locally — don't commit the spike.

2. **`feat(desktop): add CbrArchiveReader`** (only if chosen solution passes spike)
   - Implements the same `ArchiveReader` interface. Skips invisible / non-image files the same way ZIP does.

3. **`test(desktop): CbrArchiveReader round-trips a fixture`**

### Non-goals in C

- Writing CBR. We read only.

## Slice D — Folder scanner

**Goal:** given a user-selected root, produce a `ScanResult` that the UI can
review before committing. Never touches the DB.

### Phases

1. **`feat(desktop): add LocalScannerService`**
   - `modules/local/scanner/scanner.service.ts`.
   - Walks the root with a bounded concurrency worker pool (default 4). Skips `.`/`_` prefixed directories and typical noise (`@eaDir`, `.DS_Store`, `Thumbs.db`).
   - Two-level detection heuristic:
     - **Flat layout** (`Root/SeriesName/chapter.cbz`): each direct child is a series; each archive/subfolder inside is a chapter.
     - **Nested layout** (`Root/SeriesName/Volume 01/Chapter 01/001.jpg`): if a child folder has no image files directly but its grandchildren do, each child = series, grandchildren = chapters.
     - **Single-series layout** (`Root/001.cbz`, `Root/002.cbz` directly): root is one series, each archive is a chapter.
   - For each candidate chapter, read the first page via `ArchiveReader` and sample the first entry name for cover-candidate selection.
   - Chapter/volume number inference: regex `\b(?:ch|chapter|c)[\s_.-]*([0-9]+(?:\.[0-9]+)?)\b` case-insensitive; volume: `\b(?:vol|volume|v)[\s_.-]*([0-9]+)\b`. Unmatched chapters get `null` and sort by name.

2. **`test(desktop): scanner detects flat, nested, and single-series layouts`**
   - Fixtures under `apps/desktop/src/modules/local/scanner/__fixtures__/`. sql.js not involved — pure fs logic.

3. **`feat(desktop): stream scan progress over socket`**
   - Gateway emits `LocalEvents.SCAN_PROGRESS` every 200ms (debounced) with `ScanProgress`.
   - `LocalScanPayload` returns the final `ScanResult` as the ack response.

### Non-goals in D

- Actually writing to the DB. Everything produced here is proposal-only.

## Slice E — Import flow

**Goal:** the user sees the scan proposal, can deselect series/chapters, edits
titles if they want, and commits. The desktop writes series+chapter rows and
emits a single `library:updated` so the existing library store re-fetches.

### Phases

1. **`feat(desktop): LocalLibraryService.import()`**
   - Transactional: for each confirmed `ScanCandidateSeries`, insert a `series` row with `source='local'`, `local_root_path`, computed `local_content_hash`, and default `status='reading'`. Insert `chapters` rows with `local_path` + `local_archive_format`.
   - Extract cover: read first page of the first chapter via `ArchiveReader`, copy bytes to `userData/covers/local/{seriesId}/cover.jpg` atomically (tmp → rename). Write `cover_path` as `kirei-cover://local/{seriesId}/cover.jpg`.
   - Broadcast `library:updated` with `action: 'followed'` (reuse the existing event — the renderer library store already refetches on this).

2. **`feat(desktop): add LocalGateway`**
   - `SCAN`, `IMPORT`, `DELETE`, `UPDATE_SERIES`, `RESCAN` subscribe handlers. Mirrors `LibraryGateway`'s structure — `handleGatewayRequest` wrapper, ws-throttler guard.

3. **`feat(web): import flow UI — folder picker and review panel`**
   - New route `/library/import`. Triggered from the existing Library empty-state CTA (add a second button next to "Browse MangaDex": "Import Local").
   - Phase 1: `useFolderPicker()` hook → `dialog.showOpenDialog({ properties: ['openDirectory'] })` proxied over the gateway (new `LOCAL_PICK_FOLDER` or a generic `SYSTEM_PICK_FOLDER` channel — prefer the generic one so v0.3 settings can reuse it).
   - Phase 2: scan progress — editorial progress bar (hairline + bengara fill), currently-scanning path below in mono.
   - Phase 3: review table — rows of detected series with checkbox, inline-editable title, chapter count, expand-to-see-chapters toggle.
   - Phase 4: commit — calls `local:import`, navigates to `/` on success.

4. **`test(desktop): import round-trip writes series and chapters`**
   - Integration test with sql.js and a fixture folder. Assert row counts and cover file existence (mock out actual file copy — unit test the extraction separately).

### Non-goals in E

- Deleting / rescanning (own slices).
- MangaDex match (Slice K).

## Slice F — `kirei-page://local/...` protocol + cover extraction

**Goal:** the reader can fetch local pages the exact same way it fetches
MangaDex pages. Local covers are served via `kirei-cover://local/...` with the
same path-validation and caching discipline as the MangaDex branch.

### Phases

1. **`feat(desktop): extend kirei-page protocol with local branch`**
   - `kirei-page://local/{localChapterId}/{pageIndex}.{ext}`.
   - Handler branches on the first segment — existing mangadex branch unchanged.
   - Local branch: look up chapter by id → open its archive → read page at index → stream bytes with the right content-type. No disk cache needed (filesystem is already the cache).
   - Path validation: `^[a-zA-Z0-9-]+$` for `localChapterId` (must match UUID format); index must be `^\d+$`; extension from a whitelist.

2. **`feat(desktop): extend kirei-cover protocol with local branch`**
   - `kirei-cover://local/{seriesId}/cover.jpg` → reads from `userData/covers/local/{seriesId}/cover.jpg`. No fetch fallback.

3. **`feat(desktop): add local.get-pages handler`**
   - Returns an array of `kirei-page://local/{chapterId}/{i}.{ext}` URLs for the given local chapter. Opens the archive once, lists entries, closes, returns indices with extensions.

4. **`test(desktop): local page protocol serves bytes with correct mime`**
   - Integration test with a fixture CBZ.

### Non-goals in F

- Re-encoding / resizing (we serve the source bytes).

## Slice G — Local series detail page + chapter list

**Goal:** clicking a local series in the library opens the existing series
detail layout, but populated from SQLite instead of MangaDex. The same
follow/unfollow button, status chips, and chapter list all work.

### Phases

1. **`feat(web): branch SeriesDetail on source`**
   - `SeriesDetail.tsx` already reads `useMangaDexSeries(mangadexId)`. Add `useLocalSeries(localId)` + a top-level branch selector that picks the hook based on a new URL shape: `/series/local/:id` vs the existing `/series/:mangadexId` → rename existing to `/series/mangadex/:mangadexId` with a redirect for old URLs.
   - Share the layout primitives (banner, meta row, synopsis, chapter list) — only the data hook differs.

2. **`feat(desktop): local.get-series handler`**
   - Returns `{ series: Series, chapters: ChapterListItem[] }`. Maps local chapter rows to `ChapterListItem` (re-use the shape to avoid divergent UIs — `translatedLanguage` is null for local, `scanlationGroup` is null).

3. **`feat(web): chapter list reads local chapters`**
   - Existing chapter list already renders from `ChapterListItem[]`. Wire the local branch, skip the language filter dropdown when source is local.

4. **`feat(web): local status + notes editor`**
   - Status chip reuses the existing mutation. Notes get a small editor (textarea with Fraunces italic placeholder "Notes on this series…" ). Saves via `local:update-series`.

### Non-goals in G

- Editing cover / title (Slice J).

## Slice H — Local reader integration

**Goal:** opening a local chapter launches the existing reader. Page fetching,
keyboard nav, double/webtoon modes, and bookmarks all work unchanged — the
only change is the page-list source.

### Phases

1. **`feat(web): reader picks local vs mangadex page source`**
   - `useChapterPages(chapterId)` currently calls `mangadex:get-pages`. Add a branch: if the chapter row says `source === 'local'`, call `local:get-pages` instead.
   - The reader route already uses the local chapter id in the URL for mangadex chapters (it's stored in the router state) — for local we pass the local UUID.

2. **`feat(desktop): track progress for local chapters`**
   - `reader:update-progress` and `reader:mark-read` accept local chapter id (add `source` to the payload or use a discriminator). The existing upsert path keys on `id` directly for local rows instead of `mangadex_chapter_id`.
   - Emit the same `library:updated` event.

3. **`feat(web): resume from last read works for local series`**
   - The existing "Continue" button on series detail already reads `lastChapterId`. For local, it points to a local chapter id. Navigation is unchanged.

4. **`test(desktop): local reader progress round-trips`**

### Non-goals in H

- Translation overlay (v0.3).

## Slice I — Unified library view

**Goal:** the Library grid and list views show local + MangaDex series together.
A small "source" badge identifies local entries (bengara outline ring vs.
filled for MangaDex, or a leading `編` kanji glyph — pick one via spike).

### Phases

1. **`feat(web): source badge on cover card`**
   - Subtle — the library should not feel like two parallel worlds.

2. **`feat(web): source filter in library controls`**
   - Existing status / sort filter row gets a new segmented control: `All / MangaDex / Local`.
   - Saves to `browse-store` or a new `library-store` slice if it doesn't already exist.

3. **`feat(web): library search covers both sources`**
   - Existing fuzzy search is client-side over the full library entries — no new work, but verify the UX still feels right at 200+ entries.

### Non-goals in I

- Cross-source deduplication (if a user imports a local copy of a MangaDex series, they get two entries; merging is an opt-in action from Slice K).

## Slice J — Manual metadata editor

**Goal:** edit title, cover, score, status, notes, volume/chapter numbers on
local series and chapters without a MangaDex round-trip.

### Phases

1. **`feat(web): series metadata editor drawer`**
   - From the series detail meta row, a pencil icon opens a right-docked drawer. Fields: title, titleJapanese, status, score (1–10), notes, cover (upload a new image file via file picker).
   - Cover replacement: new file → copy into `userData/covers/local/{seriesId}/cover.{ext}`, update `cover_path`.

2. **`feat(web): chapter list inline edits`**
   - Edit chapter number / volume number / title via a small popover on each chapter row. Useful for mis-parsed folder names.

3. **`feat(desktop): local.update-series and local.update-chapter handlers`**
   - Thin wrappers around SQLite UPDATE with a whitelist of editable columns.

4. **`test(desktop): metadata edits persist`**

## Slice K — Optional MangaDex metadata match

**Goal:** a user with a local series can optionally look up the same series on
MangaDex to pull in synopsis, genres, author, official covers, and (crucially)
the MangaDex id so update checks work for ongoing local reads.

### Phases

1. **`feat(web): "Find on MangaDex" action on local series`**
   - Button in the metadata editor drawer. Runs `mangadex:search` with the current title. Shows top 5 results with cover + title + year; user picks or cancels.

2. **`feat(desktop): attach mangadexId to a local series`**
   - Updates `series.mangadex_id` (UNIQUE — must fail gracefully if already attached to another row).
   - Optionally overwrites title / titleJapanese / notes / synopsis if user opts in (checkboxes in the picker).

3. **`feat(web): show attached MangaDex metadata on local series`**
   - When a local series has a `mangadexId`, the series detail page fetches and displays synopsis / genres / author from MangaDex (the local chapter list still drives reading).

### Non-goals in K

- Merging two library entries into one (defer).

## Slice L — Update checks for folders

**Goal:** when a followed local series' root folder gains a new chapter
folder/archive, it surfaces as an update badge just like MangaDex new-chapter
notifications.

### Phases

1. **`feat(desktop): local update poller`**
   - Runs `LocalScannerService` narrowed to each followed local series' `local_root_path`. Diffs against existing `chapters` rows (by relative path), inserts new rows, bumps `new_chapter_count`.
   - Reuses the same `library:updates-available` event the mangadex poller uses.

2. **`feat(desktop): on-demand rescan from series detail`**
   - Button in local series detail meta row: "Rescan". Triggers the above for just that series.

3. **`test(desktop): rescan detects new chapters, ignores existing`**

### Non-goals in L

- Filesystem watchers (fs.watch on Windows is flaky across network drives). Poll-based is fine for v0.2.

## Open questions

- **CBR format:** ship CBZ-only in v0.2 and add CBR in v0.2.1, or block v0.2 on CBR? Leaning CBZ-only — existing collections can be converted, and the unrar licensing story is messy.
- **Root strategy:** one root per library, multiple roots, or "library folders" concept? Leaning one root for v0.2 — multi-root adds DB complexity that doesn't match v0.2's scope.
- **Archive caching:** re-extract on every reader open, or cache extracted pages under `userData/pages/local/{chapterId}`? Leaning re-extract. Disk cost of a cache is real; file-read speed from a local archive is fine.
- **Natural sort locale:** `Intl.Collator('en', { numeric: true })` or a hand-rolled comparator? Leaning Intl — free, portable, handles leading zeros.
- **Cover extraction on import vs on demand:** eagerly at import so the library renders instantly, or lazily in the cover protocol? Leaning eager — import is a one-time cost, library grid is hot.
- **Local chapter URL exposed in reader:** `/reader/:localId` or `/reader/local/:localId`? The existing route is `/reader/:chapterId`; local ids are UUIDs that won't collide with MangaDex UUIDs so a flat namespace is fine. Decide during Slice H.
