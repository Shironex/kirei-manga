# KireiManga v0.4 — AniList Sync

Tracks the work for v0.4 (AniList Sync) as defined in `docs/kireimanga-prd.md` §5.4.
Reuses the AniList OAuth pattern already battle-tested in ShiroAni — the
intent is to share as much of the code as possible via a future
`@shiro/anilist-client` package, but for v0.4 we inline a minimal client to
avoid blocking on cross-app packaging.

## Cross-cutting rules

- Same repo, same working-tree rules as prior milestones. No `pnpm dev` without user OK. No `Co-Authored-By`.
- AniList API requires an OAuth implicit-grant token (no PKCE for desktop — spec uses implicit grant). Token lives for 1 year by default.
- Rate limit: AniList allows ~90 req/min per token (official-ish — they throttle by IP+token). Treat anything above 1 req/second as worth queueing. The gateway pattern from the MangaDex client applies — one gate, 700ms interval, with 429 retry.
- AniList ids are integers; keep the domain type as `number` (never stringify in SQLite).
- AniList "CHAPTERS_READ" is an integer (latest read chapter, rounded down). If the user just read chapter 42.5, sync pushes 42. Decimal chapters never round up.
- Never sync progress that goes backwards without user confirmation (e.g. a re-read). Store `anilist_last_pushed_chapter` to detect regressions.
- Offline-safe: all sync actions are queued to a local outbox; flush on next online event.
- The user can disable AniList sync entirely in settings — the rest of the library behaves as if the feature isn't installed.
- Design identity per `.impeccable.md`: AniList status/score appears as a small hairline-outlined badge next to the in-app status, never replacing it.

## Slice A — Shared types for AniList entities

**Goal:** types the gateway, services, and renderer all consume. Kept minimal —
AniList's GraphQL schema has many fields but we only want a small subset.

### Phases

1. **`feat(shared): add anilist types`**
   - `packages/shared/src/types/anilist.ts`:
     - `AniListStatus = 'CURRENT' | 'PLANNING' | 'COMPLETED' | 'DROPPED' | 'PAUSED' | 'REPEATING'` (AniList spelling).
     - `AniListManga` → `{ id: number; title: { romaji: string; english?: string; native?: string }; coverImage: { large: string }; chapters?: number; siteUrl: string }`.
     - `AniListEntry` → `{ id: number; mediaId: number; status: AniListStatus; score: number; progress: number; updatedAt: number; media: AniListManga; notes?: string }`.
     - `AniListSyncState = 'idle' | 'syncing' | 'error' | 'offline'`.
     - Mapper: `aniListStatusToReadingStatus(s): ReadingStatus` and the reverse, in a small `constants/anilist.ts` module.

2. **`feat(shared): add anilist channel payloads`**
   - `ipc-payloads.ts`:
     - `AniListLoginResponse` (`{ success; reason? }` — token is never sent to the renderer).
     - `AniListLogoutResponse`.
     - `AniListGetAccountResponse` (`{ user: { id; name; avatar? } | null }`).
     - `AniListSyncStatusEvent` (`{ state; lastSyncedAt?; pendingOps; error? }`).
     - `AniListImportListResponse` (`{ imported; skipped; linkedExisting }`).
     - `AniListLinkSeriesPayload/Response`, `AniListUnlinkSeriesPayload/Response`.
     - `AniListPushProgressPayload/Response` (manual flush trigger for debugging).
   - `constants/ipc-events.ts` → `AniListEvents` enum.

3. **`feat(desktop): migration 009 for anilist state`**
   - `ALTER TABLE series ADD COLUMN anilist_entry_id INTEGER` (distinct from `anilist_id` — `anilist_id` is the _media_ id, `anilist_entry_id` is the user's list entry id; both matter for mutations).
   - `ALTER TABLE series ADD COLUMN anilist_last_pushed_chapter REAL`.
   - `ALTER TABLE series ADD COLUMN anilist_last_synced_at TEXT`.
   - `CREATE TABLE anilist_outbox (id TEXT PRIMARY KEY, series_id TEXT NOT NULL REFERENCES series(id) ON DELETE CASCADE, op TEXT NOT NULL CHECK (op IN ('push-progress','update-status','update-score')), payload TEXT NOT NULL, created_at TEXT NOT NULL, attempts INTEGER NOT NULL DEFAULT 0, last_attempt_at TEXT, last_error TEXT);`
   - Index on `(series_id)` + `(created_at)`.

## Slice B — OAuth flow + token storage

**Goal:** user clicks "Connect AniList", app opens a dedicated Electron window
at AniList's auth URL, captures the resulting `access_token` from the redirect
fragment, stores it in `electron-store` under an encrypted key.

### Phases

1. **`feat(desktop): anilist auth service scaffold`**
   - `modules/anilist/auth/anilist-auth.service.ts`:
     - `startLogin()`: opens a BrowserWindow at `https://anilist.co/api/v2/oauth/authorize?client_id=X&response_type=token` (implicit grant). Uses `will-redirect` / `will-navigate` to catch the redirect to our desktop redirect URI (a local loopback is easiest — reuse the custom protocol if simpler).
     - `logout()`: clears token, emits `anilist:sync-status` with `state='idle'`.
     - `getAccessToken()`: internal — never exposed over the socket.

2. **`feat(desktop): encrypted token storage`**
   - Use `safeStorage.encryptString(token)` (Electron built-in, backed by OS keychain on macOS and DPAPI on Windows) — write ciphertext into `electron-store`.
   - On read, `safeStorage.decryptString`. If safeStorage is unavailable, fall back to plain with a warning log and a settings badge.

3. **`feat(desktop): anilist client id configuration`**
   - `packages/shared/src/constants/anilist.ts` exports the client id. Create a dedicated AniList application for KireiManga (separate from ShiroAni — different redirect URI and scopes).
   - Document redirect URI in the roadmap.

4. **`feat(web): settings panel — AniList connection`**
   - New row under settings. Connect button → calls `anilist:login`. Connected state shows avatar + username + "Disconnect".
   - Sync status pill matching the editorial badge style.

5. **`feat(desktop): anilist:get-account handler`**
   - Resolves `Viewer { id, name, avatar.medium }` via GraphQL. Returns null when no token.

### Non-goals in B

- Token refresh — AniList implicit-grant tokens last 1 year. When expired, prompt re-login.

## Slice C — GraphQL client + rate-limit handling

**Goal:** a thin GraphQL client with a single-gate rate-limit queue and 429
retry. Mirrors `MangaDexClient` in discipline but simpler (only one endpoint,
one auth header).

### Phases

1. **`feat(desktop): anilist graphql client`**
   - `modules/anilist/anilist.client.ts`:
     - Endpoint: `https://graphql.anilist.co`.
     - `query<T>(gql, variables)` and `mutate<T>(gql, variables)` — both serialize through a single `generalGate` (700ms min interval).
     - 429 handling: respect `Retry-After`, max 3 retries with exponential backoff.
     - Token injected on every request; missing token returns `{ ok: false, reason: 'not-logged-in' }`.

2. **`feat(desktop): prebaked gql documents`**
   - `anilist/gql/` folder with `.graphql` files: `Viewer`, `MediaListCollection`, `SaveMediaListEntry`, `DeleteMediaListEntry`.
   - Compiled to string constants at build-time (simple `readFileSync` at module load is fine — no codegen).

3. **`test(desktop): anilist client enforces rate limit + retries`**

### Non-goals in C

- Persistent query caching. AniList data is small; re-fetch is fine.

## Slice D — Import user's manga list

**Goal:** first-sync pulls the user's entire AniList manga list, tries to
link each entry to an existing local series (by MangaDex id via AniList's
`Media.externalLinks` field, or fuzzy title match), imports unlinked entries
as plan-to-read placeholders.

### Phases

1. **`feat(desktop): import service — fetch AniList list`**
   - `modules/anilist/anilist-import.service.ts`:
     - Paginates `MediaListCollection(userName, type: MANGA)`.
     - For each entry: if local library has a series with matching `anilist_id`, skip (already linked). Otherwise try to resolve via AniList `Media.externalLinks` (MangaDex URLs embed the MangaDex UUID); if still unmatched, fuzzy title match against `series.title` + `series.title_japanese` with a conservative threshold (0.85 Dice coefficient).

2. **`feat(desktop): import mutation — link and/or create`**
   - Matched: `UPDATE series SET anilist_id = ?, anilist_entry_id = ?, score = COALESCE(score, ?) WHERE id = ?`.
   - Unmatched: create a placeholder row with `source='mangadex'` if MangaDex id is known, otherwise `source='local'` with no `local_root_path` (virtual placeholder — user can attach files later).
   - Outbox: no writes go to AniList during import. Import is read-only.

3. **`feat(web): import flow UI`**
   - "Import AniList list" button in settings. Modal: progress bar, "X matched / Y created / Z conflicts" summary when done.
   - Conflict list inline: each conflict is a row with `[local] vs [anilist]`, user picks which is source of truth (first version: always prefer AniList, conflict UI in Slice G).

4. **`test(desktop): import links by mangadex id and skips already-linked`**

### Non-goals in D

- Bulk merge of duplicate entries — Slice G / user-initiated.

## Slice E — Two-way progress sync

**Goal:** when the user finishes a chapter locally, the AniList entry's
`progress` (chapters read) + `status` (auto-promote PLANNING → CURRENT)
update. Local changes queue into an outbox that the sync worker drains.

### Phases

1. **`feat(desktop): outbox worker`**
   - `modules/anilist/anilist-sync.worker.ts`:
     - Polls every 30s for queued ops. Drains FIFO with AniList rate limits applied via the client.
     - On success: deletes the row, updates `anilist_last_pushed_chapter` / `anilist_last_synced_at` on the series.
     - On transient error (network, 5xx): increments `attempts`, keeps the row, retries after exponential backoff.
     - On permanent error (401, 403): pauses the worker, emits `anilist:sync-status` with `error`.

2. **`feat(desktop): enqueue on progress updates`**
   - Extend `LibraryService.updateProgress` / `markChapterRead`: if the series has an `anilist_id` and the newly-read chapter exceeds `anilist_last_pushed_chapter`, enqueue a `push-progress` op.
   - If `series.status` just transitioned to `reading` from `planToRead`, enqueue an `update-status` op to `CURRENT`.
   - If local `status` changed to `completed`, enqueue `update-status` to `COMPLETED` and push final chapter count.

3. **`feat(desktop): pull remote updates periodically`**
   - Every 15 minutes (and on app focus), fetch `MediaList(userId, mediaId_in: [...followedIds])` and reconcile: if remote progress > local, mark chapters 1..remoteProgress as read locally (with a confirmation dialog for a gap larger than 3).

4. **`feat(web): sync status indicator`**
   - Footer pill near the sidebar version: "Synced · 2m ago" / "Syncing…" / "Sync paused — reconnect".

5. **`test(desktop): outbox drains in order and respects rate limit`**

### Non-goals in E

- Syncing custom lists, favourites, or reviews.

## Slice F — Display AniList status + score in library

**Goal:** users see their AniList status/score alongside local status. Hover
the AniList badge to jump to the AniList page.

### Phases

1. **`feat(web): anilist badge on CoverCard`**
   - Small circle in the corner showing the status color (CURRENT bengara, COMPLETED kinari, PLANNING sumi). Tooltip shows "AniList: Current · 7.8".

2. **`feat(web): series detail meta row shows anilist score`**
   - Alongside local score. Editable via the metadata drawer; edits enqueue an `update-score` op.

3. **`feat(web): library filter — linked / unlinked`**
   - "AniList: linked / unlinked / all" segmented control.

### Non-goals in F

- Syncing notes / reviews (AniList notes are free-text; keep separate from local notes for v0.4).

## Slice G — Conflict resolution UI

**Goal:** when local and remote progress diverge enough to matter, the user
sees a small resolution panel instead of a silent auto-merge.

### Phases

1. **`feat(web): conflict inbox in settings`**
   - List of pending conflicts. Each row: series cover + title + "Local: ch 12 · AniList: ch 7" + three buttons: `Keep local`, `Keep AniList`, `Ignore`.

2. **`feat(desktop): conflict detection`**
   - During the periodic pull (Slice E step 3), any divergence > 3 chapters or a status mismatch enqueues into a `conflicts` table.

3. **`feat(desktop): conflict resolution handler`**
   - Applies the user's choice: push local / pull remote / dismiss.

### Non-goals in G

- Auto-merge policies beyond "prefer higher" or "prefer most recent" — keep choices explicit.

## Slice H — Cross-source linking helper

**Goal:** a local-imported series and a MangaDex-followed entry of the same
title can end up attached to the same AniList id. Detect and offer to merge
library entries.

### Phases

1. **`feat(desktop): duplicate-by-anilist-id detector`**
   - Query `SELECT anilist_id, COUNT(*) FROM series WHERE anilist_id IS NOT NULL GROUP BY anilist_id HAVING COUNT(*) > 1`.
   - Surface as a settings "Library health" panel row.

2. **`feat(web): merge duplicates flow`**
   - User picks the surviving row; chapters from the other row are re-parented (if their ids don't conflict) or left orphaned with a confirmation.
   - Bookmarks + reading sessions follow the surviving series.

### Non-goals in H

- Automated merging without user confirmation.

## Open questions

- **Redirect URI:** Electron desktop apps typically use a loopback (`http://127.0.0.1:PORT/callback`) or a custom protocol (`kireimanga://auth/anilist`). Custom-protocol feels cleaner; loopback is less flaky. ShiroAni uses custom-protocol — lean that way.
- **GraphQL codegen:** worth generating types from the AniList schema? At v0.4's scale, inline typed query functions are fine. Revisit if we hit 10+ queries.
- **Chapter float rounding:** AniList `progress` is an integer, local chapter numbers are floats (`42.5`). Floor on push is the AniList-standard convention. Document loudly.
- **Token expiry handling:** proactive re-auth prompt at 30 days remaining, or wait for 401? Reactive is less annoying for most users. Go reactive + a one-line warning in settings near expiry.
- **Sync on startup:** always fetch viewer + sync status at boot, or only on focus? Boot-once then focus-driven. Keeps cold-start light.
